# 토스 미니앱 공통 개발 가이드

## 목적과 범위

이 문서는 이 워크스페이스의 토스 미니앱 프로젝트에서 공통으로 사용하는 개발 기준을 정리한 문서입니다.

- 대상 프로젝트: `point-cafe`, `point-cat`, `point-diamond`, `point-pig`
- 포함 범위: 프로젝트 구조, TDS 활용, Apps in Toss 연동, Firebase, 분석, QA, 배포
- 제외 범위: 포인트 적립 정책, 적립 수치, 프로모션 보상 로직, 경제성 밸런싱

핵심 원칙은 다음과 같습니다.

- 앱별 브랜딩과 도메인 로직은 달라도 개발 뼈대는 최대한 동일하게 유지합니다.
- 토스 앱 SDK 호출은 페이지에서 직접 흩뿌리지 않고 공통 래퍼로 모읍니다.
- UI는 TDS를 기본값으로 사용하고, 프로젝트 공통 래퍼를 먼저 재사용합니다.

## 공통 기술 스택

- `Vite + React 18 + TypeScript`
- `@toss/tds-mobile`, `@toss/tds-mobile-ait`
- `@apps-in-toss/web-framework` 기반 Apps in Toss 설정
- `Firebase Firestore`
- `PostHog`
- `ESLint`, `Prettier`

현재 프로젝트들은 모두 비슷한 구조를 따릅니다.

- `granite.config.ts`: Apps in Toss 메타 정보와 로컬 개발 설정
- `src/main.tsx`: 앱 진입점, TDS Provider, analytics 초기화
- `src/env.ts`: 런타임 설정과 환경 변수 fallback
- `src/firebase/init.ts`: Firebase 초기화
- `src/lib/tossSdk.ts`: 토스 앱 브릿지 래퍼
- `src/lib/analytics.ts`: PostHog 공통 추적 함수
- `src/components/*`: TDS 래퍼와 공통 UI 컴포넌트
- `firestore.rules`, `firestore.indexes.json`: Firestore 보안 규칙과 인덱스

## 시작하기

프로젝트별 개발 시작 절차는 아래 순서를 기본으로 합니다.

1. 프로젝트 폴더로 이동합니다.
2. 의존성을 설치합니다.
3. `.env.example`를 기준으로 `.env.local`을 구성합니다.
4. 로컬 개발 서버를 실행합니다.


배포 스크립트는 `point-cafe`, `point-cat`, `point-pig`처럼 `npm run deploy` 형태로 통일하는 것을 권장합니다.

## 공통 파일 책임

### 1. `granite.config.ts`

토스 미니앱 메타 정보의 단일 진입점입니다.

- `appName`: 콘솔/앱 식별용 이름
- `brand.displayName`: 사용자 노출 앱명
- `brand.primaryColor`: 대표 색상
- `brand.icon`: 앱 아이콘 URL
- `web.host`, `web.port`: 로컬 개발 서버 설정
- `navigationBar`: 기본 내비게이션 정책

새 앱을 만들거나 기존 앱 브랜딩을 바꿀 때는 먼저 여기부터 수정합니다.

### 2. `src/main.tsx`

앱 진입점은 반드시 아래 역할만 담당하도록 유지합니다.

- TDS 색상 CSS import
- `TDSMobileProvider` 설정
- `initAnalytics()` 실행
- `<App />` 마운트

진입점에 페이지 로직이나 도메인 조건문을 넣지 않습니다.

### 3. `src/env.ts`

환경 변수는 여기서 읽고, 페이지에서는 `runtimeConfig`만 사용합니다.

- 광고 그룹 ID
- 공유용 딥링크/OG 정보
- PostHog 설정
- 기타 런타임 상수 fallback

페이지나 컴포넌트에서 `import.meta.env`를 직접 읽는 패턴은 늘리지 않는 편이 좋습니다.

### 4. `src/firebase/init.ts`

Firebase 초기화는 이 파일 한 곳에서만 수행합니다.

- `initializeApp()` 호출 위치를 단일화합니다.
- `getApps()`로 중복 초기화를 막습니다.
- `db` export만 노출하고, 실제 쿼리/쓰기 로직은 다른 계층으로 분리합니다.

### 5. `src/lib/tossSdk.ts`

토스 앱 전용 기능은 이 래퍼를 통해서만 접근합니다.

- 토스 앱 여부 감지
- 브라우저 개발 환경 fallback
- 로그인/유저 식별키 처리
- 공유 링크 호출
- 광고 초기화 및 노출
- 뒤로가기 이벤트 등록
- 햅틱 등 디바이스 상호작용

새 SDK 기능이 필요하면 페이지에서 바로 SDK를 호출하지 말고, 먼저 이 파일에 공통 함수로 추가합니다.

### 6. `src/lib/analytics.ts`

이벤트 수집은 공통 함수로 모읍니다.

- `initAnalytics()`
- `identifyAnalytics()`
- `track()`
- `trackAppOpened()`
- `trackScreenViewed()`

페이지에서 `posthog.capture()`를 직접 호출하기보다 이 래퍼를 먼저 사용합니다.

## TDS 활용 가이드

이 문서에서 가장 중요한 영역입니다. 현재 프로젝트들은 모두 TDS를 기본 UI 레이어로 사용하고 있고, 여기서 일관성이 깨지면 앱 완성도가 가장 크게 떨어집니다.

### 1. 필수 셋업

`src/main.tsx`에는 아래 두 가지가 반드시 들어가야 합니다.

```tsx
import { TDSMobileProvider } from "@toss/tds-mobile";
import "@toss/tds-colors/colors.css";
import "@toss/tds-colors/colors.light.css";

const browserUserAgent = {
    fontA11y: undefined,
    fontScale: 1,
    isAndroid: false,
    isIOS: true,
    colorPreference: "light" as const,
    safeAreaBottomTransparency: "opaque" as const,
};

<TDSMobileProvider userAgent={browserUserAgent}>
    <App />
</TDSMobileProvider>;
```

정리하면 다음과 같습니다.

- `colors.css`: 기본 색상 토큰 로드
- `colors.light.css`: `--adaptive*` 계열 변수 로드
- `TDSMobileProvider`: TDS 컴포넌트가 기대하는 컨텍스트 제공

이 셋업이 빠지면 BottomSheet, 배경색, 텍스트 색상, safe area 처리에서 바로 문제가 납니다.

### 2. 왜 `colors.light.css`가 중요한가

TDS 컴포넌트는 `--adaptiveLayeredBackground`, `--adaptiveBackground` 같은 CSS 변수를 사용합니다. 이 값은 `colors.light.css` 또는 `colors.dark.css`에서 공급됩니다.

따라서 `colors.light.css`가 빠지면 다음 증상이 발생할 수 있습니다.

- BottomSheet 패널 배경이 투명하게 보임
- dimmer와 패널 경계가 흐려짐
- 일부 텍스트/배경이 의도와 다르게 렌더링됨
- `getComputedStyle(...).backgroundColor`가 투명값으로 나옴

현재 프로젝트에서 BottomSheet 배경 이슈를 피하기 위해 별도 셋업 가이드가 있는 이유도 이 문제 때문입니다.

### 3. Provider를 반드시 루트에 두는 이유

`TDSMobileProvider`는 단순 테마 래퍼가 아닙니다. 브라우저 개발 환경에서 토스 앱과 유사한 UI 컨텍스트를 만들어 주는 역할까지 담당합니다.

- 폰트 스케일
- iOS/Android 추정값
- 색상 선호도
- safe area 처리 기준

특히 WebView 기반 미니앱은 safe area와 overlay 계열 컴포넌트가 민감하므로, 페이지 단위로 Provider를 중복 선언하지 말고 앱 최상단에 한 번만 둡니다.

### 4. TDS는 직접 쓰기보다 공통 래퍼를 먼저 봅니다

현재 프로젝트에는 `src/components/TdsWebView.tsx`가 있고, 여기서 TDS 컴포넌트를 프로젝트 규칙에 맞게 감싼 공통 레이어를 제공합니다.

대표적으로 아래 성격의 래퍼가 이미 존재합니다.

- 텍스트: `Txt`
- 간격: `Spacing`
- 버튼: `TdsButton`
- 아이콘/이미지: `AssetIcon`, `AssetImage`, `AppBrandMark`
- 레이아웃성 리스트: `TdsGridList`, `TdsGridListItem`
- 툴팁/오버레이: `TooltipBubble`

페이지에서는 먼저 이 공통 래퍼를 사용하고, 정말 필요한 경우에만 원본 TDS 컴포넌트를 직접 불러옵니다.

이유는 분명합니다.

- 브랜딩 색상 매핑이 이미 들어가 있습니다.
- typography 이름과 weight 규칙이 통일돼 있습니다.
- 접근성 역할과 키보드 조작까지 일부 래퍼가 보완합니다.
- 프로젝트마다 같은 버튼을 다른 스타일로 다시 만드는 중복을 줄일 수 있습니다.

예시는 아래처럼 단순해야 합니다.

```tsx
import { Spacing, TdsButton, Txt } from "./components/TdsWebView";

export function ExampleSection() {
    return (
        <section>
            <Txt typography="title" weight="bold">
                오늘의 미션
            </Txt>
            <Spacing size={12} />
            <Txt color="secondary">TDS 래퍼를 먼저 사용합니다.</Txt>
            <Spacing size={16} />
            <TdsButton tone="primary" onClick={() => {}}>
                확인
            </TdsButton>
        </section>
    );
}
```

### 5. Overlay 계열은 공통 컴포넌트를 우선 사용합니다

`BottomSheet`와 `Toast`는 원본 TDS 컴포넌트를 그대로 쓰지 않고 프로젝트 공통 래퍼를 먼저 씁니다.

이유는 다음과 같습니다.

- BottomSheet는 기본형과 fullscreen형의 동작이 이미 정리돼 있습니다.
- dim 처리 여부와 body attribute 제어가 래퍼에 들어 있습니다.
- Toast는 top/bottom 위치 처리와 mount/unmount 타이밍을 보완합니다.

실무 기준은 아래처럼 가져가면 됩니다.

- 바텀시트: `src/components/BottomSheet.tsx`
- 토스트: `src/components/Toast.tsx`
- 새 overlay 패턴이 필요하면 페이지에서 직접 구현하지 말고 공통 컴포넌트로 승격합니다.

### 6. 브랜딩은 상수에서 관리합니다

브랜드 컬러는 각 프로젝트의 `src/constants.ts`에서 관리하고, TDS 래퍼가 이를 소비하도록 유지합니다.

- `BRAND_COLOR`
- `BRAND_PRIMARY`

페이지 컴포넌트 안에서 브랜드 색상 hex 값을 직접 반복해서 쓰기 시작하면, 앱별 테마 변경 시 비용이 급격히 커집니다.

브랜드 변화가 필요한 경우 순서는 아래가 맞습니다.

1. `constants.ts` 수정
2. 공통 래퍼의 color map 확인
3. 개별 페이지 보정

### 7. TDS 사용 규칙

- 텍스트는 먼저 `Txt`를 사용합니다.
- 수직 간격은 `Spacing` 또는 공통 레이아웃 컴포넌트로 해결합니다.
- 버튼은 먼저 `TdsButton`을 사용합니다.
- 리스트/셀 형태 UI는 `List`, `ListRow`, `GridList` 계열을 우선 검토합니다.
- 아이콘/이미지는 TDS `Asset` 계열을 우선 사용합니다.
- 원본 TDS 컴포넌트를 직접 쓰는 경우, 왜 래퍼로 충분하지 않은지 확인하고 시작합니다.

### 8. TDS 관련 자주 나는 문제

#### BottomSheet 배경이 투명함

가장 먼저 확인할 것:

- `@toss/tds-colors/colors.css` import 여부
- `@toss/tds-colors/colors.light.css` import 여부
- `TDSMobileProvider` 적용 여부

#### 브라우저와 토스 앱에서 간격/overlay 체감이 다름

가장 먼저 확인할 것:

- Provider의 `userAgent` 값
- safe area를 무시하는 커스텀 CSS 존재 여부
- 원본 TDS 컴포넌트를 래퍼 없이 직접 쓰고 있지 않은지 여부

#### 앱별 UI 톤이 미세하게 달라짐

가장 먼저 확인할 것:

- 페이지 안에서 개별 hex 값이 늘어나지 않았는지
- `Txt`, `TdsButton`, `BottomSheet`, `Toast` 재사용이 깨지지 않았는지
- 공통 래퍼를 우회한 ad-hoc CSS가 늘지 않았는지

### 9. 다크 모드 대응 원칙

현재 코드 기준 기본값은 라이트 모드입니다. 다크 모드를 지원해야 한다면 아래를 함께 맞춰야 합니다.

- `@toss/tds-colors/colors.dark.css` import
- `browserUserAgent.colorPreference` 관리 방식 정리
- 커스텀 CSS가 TDS adaptive 변수와 충돌하지 않는지 검토

다크 모드를 부분적으로 흉내 내는 식의 임시 CSS 패치는 피합니다.

## Apps in Toss 연동 규칙

토스 앱 연동은 페이지에서 직접 SDK를 호출하는 방식이 아니라 `src/lib/tossSdk.ts` 래퍼 중심으로 가져갑니다.

### 기본 원칙

- 토스 앱 여부 감지는 래퍼에 맡깁니다.
- 브라우저 개발 환경에서는 fallback/mock을 허용합니다.
- 로그인, 공유, 광고, back event는 공통 함수로 접근합니다.
- 새 기능이 필요하면 래퍼를 확장하고 페이지는 그 함수만 사용합니다.

이 패턴을 유지하면 로컬 브라우저 개발과 토스 앱 실제 동작 간 차이를 한 곳에서 통제할 수 있습니다.

## 환경 변수 관리

각 프로젝트는 `.env.example`를 기준으로 `.env.local`을 구성합니다.

공통으로 많이 쓰는 범주는 아래와 같습니다.

- Firebase 웹 앱 설정
- 공유용 딥링크/OG 이미지/문구
- 광고 그룹 ID
- PostHog 프로젝트 키와 host

환경 변수 운영 규칙은 아래를 따릅니다.

- `import.meta.env` 접근은 `src/env.ts` 또는 초기화 파일로 모읍니다.
- 새 키를 추가하면 `src/vite-env.d.ts`에도 타입을 함께 추가합니다.
- 문서용 샘플은 `.env.example`에, 실제 로컬 값은 `.env.local`에 둡니다.
- PostHog 수집 키는 브라우저 수집용 `phc_` 키를 사용합니다.

`phx_`로 시작하는 개인용 키를 넣으면 이벤트 수집이 실패하므로 주의합니다.

## Firebase 가이드

현재 프로젝트들은 앱별 Firebase 프로젝트를 분리하는 방향으로 관리하고 있습니다. 공통 기준은 아래와 같습니다.

- 웹 앱 설정은 `.env.local`에서 주입
- 초기화는 `src/firebase/init.ts` 한 곳에서만 수행
- Firestore 규칙은 루트의 `firestore.rules` 사용
- 인덱스는 `firestore.indexes.json` 사용
- 배포 설정은 `firebase.json`에서 참조

배포 명령 예시는 아래와 같습니다.

```bash
firebase deploy --only firestore --project <firebase-project-id>
```

주의할 점은 다음과 같습니다.

- Firebase config를 페이지 안에서 직접 만들지 않습니다.
- `db`만 공통으로 export하고 데이터 접근 로직은 분리합니다.
- 규칙과 인덱스를 바꿨다면 앱 배포와 별도로 Firestore 배포도 수행합니다.

## 분석 이벤트 가이드

분석은 `src/lib/analytics.ts` 기준으로 통일합니다.

### 원칙

- `initAnalytics()`는 앱 시작 시 한 번만 호출합니다.
- 사용자 식별은 `identifyAnalytics()`를 사용합니다.
- 화면 진입은 `trackScreenViewed()`를 우선 사용합니다.
- 일반 이벤트는 `track()`로 보냅니다.

### 이벤트 설계 기준

- 이벤트 이름은 짧고 안정적으로 유지합니다.
- 같은 의미의 이벤트를 화면마다 다른 이름으로 만들지 않습니다.
- URL 유입 파라미터는 공통 유틸에서 수집하므로 중복 수집을 피합니다.

## QA 체크리스트

개발 완료 후 최소 아래 항목은 확인합니다.

### 브라우저 개발 환경

- 로컬 진입 시 화면이 깨지지 않는지
- TDS 색상과 텍스트가 정상 렌더링되는지
- BottomSheet와 Toast가 정상 노출되는지
- `.env.local` 누락 시 오류가 어디서 나는지 파악 가능한지
- PostHog 키가 없을 때 앱이 치명적으로 깨지지 않는지

### 토스 앱 또는 샌드박스 환경

- safe area가 정상인지
- 뒤로가기 이벤트가 의도대로 동작하는지
- 공유 기능이 실제로 열리는지
- 광고 지면이 설정된 경우 노출 이벤트가 정상인지
- navigation bar 설정이 콘솔/브릿지 정책과 충돌하지 않는지

### 디자인/구현 일관성

- 페이지마다 버튼 모양이 달라지지 않는지
- 브랜드 컬러가 상수와 공통 래퍼를 통해서만 쓰이는지
- ad-hoc CSS가 TDS 규칙을 덮어쓰지 않는지

## 배포 기준

배포 전에는 아래를 확인합니다.

1. `granite.config.ts`의 앱명, 브랜드, 아이콘, 포트가 맞는지 확인
2. `npm run build` 통과 확인
3. 필요 시 `npm run deploy` 실행
4. Firestore 규칙/인덱스 변경 시 Firebase 배포 수행
5. 콘솔 API 키, 딥링크, 광고 그룹 ID 등 운영값 점검

## 권장 작업 방식

마지막으로, 공통 코드 품질을 유지하려면 아래 방식을 기본값으로 삼는 것이 좋습니다.

- 새 UI를 만들 때는 먼저 TDS 조합으로 해결 가능한지 본다.
- 반복되는 UI 패턴은 페이지에 복붙하지 말고 공통 컴포넌트로 올린다.
- 토스 SDK 기능은 래퍼를 확장한 뒤 페이지에서 쓴다.
- 환경 변수는 `env.ts`와 `vite-env.d.ts`를 함께 수정한다.
- 브랜딩 변경은 `granite.config.ts`, `constants.ts`, 공통 래퍼 순서로 반영한다.

이 기준을 지키면 앱별 차이는 브랜드와 콘텐츠에만 남고, 개발/운영 비용은 공통 베이스 위에서 관리할 수 있습니다.
