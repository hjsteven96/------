Chapter 00

3주 만에 50만 원을 벌어다 준 미니앱

"미니앱으로 진짜 수익을 낼 수 있을까요?"

거창한 아이디어나 대규모 코딩 지식이 없어도 괜찮습니다. 실제로 5월 10일에 출시한 미니앱이 단 3주 만에 약 57만 원(578,614원)의 매출을 달성했습니다.

현재는 총 10개의 미니앱을 직접 운영하며 꾸준하게 들어오는 광고 수익 파이프라인을 구축해 나가고 있습니다.

실제 운영 중인 미니앱의 5월 한 달간 AdMob 광고 수익 인증

이 책은 단순한 개념 요약본이 아닙니다. 비개발자와 1인 창업가가 AI 도구를 활용해 7일 만에 기획부터 배포, 그리고 실제 수익화 장치까지 끝내는 실전 프로세스를 그대로 보여줍니다.

Chapter 01

이 책으로 할 수 있는 일

이 책은 “버튼만 누르면 자동으로 돈 번다”는 식으로 말하지 않습니다. 대신 AI를 써서 토스 미니앱 MVP를 끝까지 만드는 방법을 알려줍니다.

토스 미니앱은 일반 웹앱과 다릅니다. TDS, 샌드박스, 앱 설정, WebView 검증까지 같이 봐야 합니다. 이 책은 그 순서를 헷갈리지 않게 정리했습니다.

쉬운 설명

핵심은 간단합니다. 복붙하고, 실행하고, 막히면 다시 고치는 흐름을 끝까지 가져가는 겁니다.

하루 60~90분씩 보면 7일 안에 한 번 끝낼 수 있습니다. 완전 초보라면 10~14일 정도 잡으면 됩니다.

이 책이 잘 맞는 사람

  • 토스 미니앱으로 부업형 MVP를 만들고 싶은 사람
  • 공식 문서를 읽어도 실제 구현 순서가 감이 안 오는 사람
  • AI가 준 코드를 어디에 붙여야 하는지까지 알고 싶은 사람
  • 광고, 공유, Firebase, 배포까지 한 흐름으로 배우고 싶은 사람

이 책이 잘 안 맞는 사람

  • 수익을 보장하는 쉬운 부업 자료를 찾는 사람
  • 코드 없이 노코드만으로 토스 미니앱을 끝내고 싶은 사람
  • 플랫폼 정책, 검수, 광고 효율 검증 없이 바로 매출만 기대하는 사람

읽고 나면 할 수 있는 일

  1. 아이디어 1개를 토스 미니앱 MVP로 줄이는 법
  2. AI에게 필요한 문맥만 먼저 주는 법
  3. TDS와 샌드박스로 첫 화면을 검증하는 법
  4. Firebase, 광고, 공유, 배포를 한 흐름으로 묶는 법
Chapter 02

왜 공식 문서만으로는 자주 막힐까

공식 문서는 꼭 필요합니다. 다만 무엇부터 해야 하는지까지 대신 정해주진 않습니다.

AI도 마찬가지입니다. 문맥을 주지 않으면 그럴듯하지만 맞지 않는 코드를 내놓습니다. 이 책은 그 틈을 메우려고 만들었습니다.

이 책의 기준

문서는 규칙을 알려주고, AI는 초안을 만들어줍니다.

실제로 끝까지 가는 MVP는 실행 순서와 예외 처리에서 갈립니다.
이 전자책이 해결하려는 문제

실제로 자주 막히는 지점

  • TDS는 루트 Provider와 색상 토큰이 빠지면 overlay와 배경이 쉽게 깨진다.
  • 브라우저 개발 환경에서는 토스 SDK 호출 대신 fallback이 있어야 흐름 검증이 가능하다.
  • 보상형 광고는 load 와 show 를 분리하고 콜백 흐름을 명확히 기록해야 한다.
  • Firestore는 읽기 지연이나 실패에 대비해 local fallback 과 재동기화 전략이 필요하다.
  • 배포 뒤에는 env, 서버 함수, cron, 로그까지 운영 관점으로 다시 봐야 한다.

프롬프트 1. 수익형 미니앱 PRD 만들기

앱 아이디어를 7일 안에 출시 가능한 토스 미니앱 MVP로 줄일 때 사용합니다. 

너는 앱인토스 미니앱 PM이자 수익화 전략가야.

가상의 비게임 미니앱 "행운 노트"를 7일 안에 출시 가능한 MVP로 기획하려고 해.
사용자는 매일 운세 카드 1장을 열고, 마음에 드는 문장을 저장하고, 친구에게 공유할 수 있어.
광고는 보너스 카드 열람 보상형으로 붙이고 싶어.

[원하는 결과]
1. 서비스 한 줄 소개
2. 핵심 사용자 시나리오 3개
3. 최소 기능 범위(MVP)
4. 수익화 장치 2개
5. 화면 목록과 각 화면의 역할
6. 데이터 구조 초안
7. 첫 출시에서 제외할 기능
8. 출시 후 바로 볼 핵심 지표 5개

앱인토스 WebView 미니앱에 맞는 현실적인 범위로 정리해줘.

붙여넣을 위치 ChatGPT, Claude, Copilot Chat 어디든 좋지만 한 세션에서 계속 이어가는 편이 좋습니다.

정상 결과 MVP, 제외 기능, 수익화 장치, 측정 지표가 한 번에 정리됩니다.

이상 신호 로그인, 커뮤니티, 랭킹, 알림 센터처럼 범위가 큰 기능이 붙으면 과한 설계입니다.

재프롬프트 “앱인토스 WebView 미니앱 전제로, 7일 안에 출시 가능한 MVP만 남기고 다시 정리해줘.”

Chapter 03

시작 전에 환경부터 맞춥니다

처음 막히는 이유는 코드가 아니라 준비물입니다. 편집기, Node, Firebase, 토스 콘솔, 샌드박스가 안 맞으면 AI가 좋은 코드를 줘도 앞으로 나아가기 어렵습니다.

그래서 먼저 실습 가능한 상태를 만듭니다. 이 단계를 건너뛰면 나중에 원인을 찾기 더 어려워집니다.

시작 전에 필요한 준비물

  • 코드 편집기 또는 AI 코딩 도구 1개(Cursor, Claude Code, Codex, VS Code + Copilot 등)
  • 기본 터미널 사용법
  • Node.js LTS 와 npm
  • Firebase 계정과 새 프로젝트 1개
  • 앱인토스 콘솔 접근 권한 및 앱 검토 요청 (승인까지 2~3일 소요되므로 개발 전 신청 권장 - 이름, 아이콘, 컨셉만 작성)
  • 앱 아이콘 이미지 (ChatGPT의 DALL-E 이미지 생성 기능을 이용하면 간편함)
  • 공식 문서를 붙여 넣거나 연결할 수 있는 AI 도구 1개 이상
  • 앱인토스 샌드박스 앱 또는 실제 토스 앱 WebView 테스트 환경

어떤 AI 도구를 써도 공통으로 필요한 것

  • 파일을 직접 열고 수정할 수 있는 환경
  • 공식 문서 URL, llms.txt, llms-full.txt, TDS 문서를 AI에 연결하는 방법
  • 터미널 출력, 오류 로그, 파일 경로를 함께 보여주는 습관
  • 같은 세션에서 맥락을 이어가며 재질문할 수 있는 작업 방식

쉬운 설명

이 책의 출발점은 코딩이 아니라 실습 가능한 환경 만들기입니다.

브라우저에서 화면이 보인다고 끝이 아닙니다. 샌드박스 앱과 실제 WebView에서 다시 봐야 합니다.

현실적인 일정표

  • 경험자: 하루 60~90분 기준 7일
  • 완전 초보: 10~14일 정도로 여유 있게 잡는 편이 안전함
  • 광고, 배포, 검수 준비까지 하면 마지막 2일은 검증에 쓰는 편이 좋음

프롬프트 2. AI 툴 공통 작업환경 점검표 만들기

시작 전에 계정, 도구, 설치 항목, 첫 실행 순서를 빠르게 정리할 때 사용합니다. 

너는 앱인토스 미니앱 온보딩 코치야.

토스 미니앱을 처음 만드는 비개발자가 실습을 시작하려고 해.
Cursor, Claude Code, Codex, VS Code + Copilot 중 하나를 사용해 토스 미니앱을 만들 예정이야.
Node.js, Firebase, 앱인토스 콘솔, 샌드박스 테스트까지 준비 상태를 점검하는 체크리스트를 만들어줘.

[출력 조건]
1. 필수 계정 목록
2. 필수 설치 항목
3. 첫 실행 전에 끝내야 할 설정
4. 시작 전에 막히기 쉬운 문제 10개
5. 하루 60~90분 기준 7일 학습 계획
6. 초보자용 우선순위 순서

붙여넣을 위치 책을 읽기 전에 먼저 실행해서 준비물 체크표를 받아두세요.

정상 결과 설치, 계정, 테스트 기기, 일정표까지 한 번에 보입니다.

이상 신호 도구 목록만 길게 나오고 실제 순서가 없으면 다시 요구해야 합니다.

재프롬프트 “초보자가 바로 실행할 수 있게 순서형 체크리스트로 다시 써줘.”

처음 막히면 이렇게 봅니다

  1. 실행 로그가 있는지 본다.
  2. .env.local 과 앱 설정 파일 누락이 없는지 본다.
  3. 브라우저에서만 본 것이 아닌지 확인한다.
  4. 샌드박스 실행 조건과 기기 네트워크 상태를 확인한다.
  5. 그래도 안 되면 로그와 파일 경로를 포함해 AI에 다시 묻는다.
Chapter 04

먼저 돈 되는 MVP부터 정합니다

이 책의 예제는 행운 노트입니다. 매일 카드 한 장을 열고, 저장하고, 공유하는 간단한 앱입니다.

간단하지만 필요한 건 거의 다 들어 있습니다. 광고, 저장, 공유, OG, 재방문 설계까지 한 번에 연습할 수 있습니다.

행운 노트 기본 MVP 범위

  • 오늘의 운세 카드 1장을 여는 홈 화면
  • 문장 저장과 저장 목록 보기
  • 친구에게 공유하는 링크 버튼
  • 보너스 카드 열람을 위한 보상형 광고 지면
  • 브랜드 색상과 아이콘이 반영된 앱 설정

같은 구조로 확장 가능한 수익화 모델 3개

  • 콘텐츠형: 오늘의 운세, 오늘의 조언, 오늘의 카드처럼 매일 다시 열게 만드는 구조
  • 적립형: 일정 행동 후 광고를 보고 포인트나 보상을 받는 구조
  • 정보형: 시세, 가격, 알림형 정보를 제공하고 저장·재방문을 유도하는 구조

수익화 원칙

광고를 붙이기 전에, 왜 광고를 보는지부터 보여줘야 합니다.
수익화가 기능이 아니라 장치인 이유

이상 신호 기능이 많고 화면 수가 과도하게 늘어나면 7일 범위를 벗어난 설계입니다.

재프롬프트 “화면 4개 안쪽, 7일 MVP 기준으로 더 작게 줄여줘.”

Chapter 05

AI에게 먼저 문맥을 줍니다

AI는 질문을 잘한다고 잘 쓰는 게 아닙니다. 먼저 어떤 앱인지 정확히 알려줘야 합니다.

토스 미니앱 WebView, TDS, granite.config.ts, 샌드박스 검증. 이 네 가지는 처음부터 같이 줘야 합니다.

문서도 먼저 연결해두면 좋습니다. llms.txt, llms-full.txt, TDS 문서만 붙여도 답변 품질이 많이 달라집니다.

쉬운 설명

AI는 알아서 토스 미니앱 코드를 뽑아주지 않습니다. 먼저 여기가 토스 미니앱이라고 알려줘야 합니다.

  • granite.config.ts는 앱의 이름표입니다.
  • src/main.tsx는 TDS 와 앱 시작 설정이 모이는 곳입니다.
  • 샌드박스는 “브라우저에서 멀쩡한데 앱에서는 깨지는 문제”를 잡아내는 곳입니다.

먼저 연결할 문서

  • llms.txt: 핵심 기능 위주로 빠르게 연결할 때
  • llms-full.txt: 전체 문맥과 예시까지 넓게 참고할 때
  • TDS WebView 문서: 버튼, 리스트, Asset, Navigation, BottomCTA 같은 UI 규칙을 맞출 때
  • MCP 또는 Apps in Toss Skills: 프로젝트 구조와 문서를 더 깊게 붙일 수 있을 때

도구별로 시작하는 법

  • Cursor: Docs 인덱싱에 llms.txt, llms-full.txt, TDS 문서를 넣고 @docs를 적극 사용한다.
  • Claude Code: Apps in Toss Skills 또는 MCP를 먼저 연결하고 필요하면 claude mcp add --transport stdio apps-in-toss ax mcp start를 사용한다.

  • VS Code + Copilot: 워크스페이스에 가이드 파일, 설정 파일, 오류 로그를 같이 두고 파일 경로와 함께 질문한다.

  • granite.config.ts, src/main.tsx, 페이지 구조를 함께 설명한다.
  • 샌드박스에서 다시 확인할 포인트를 따로 적어준다.

다시 시켜야 하는 답변

Chapter 06

Firebase, 광고, 공유, 분석은 같이 봅니다

광고만 붙인다고 수익화가 되진 않습니다. 저장, 공유, 광고, 분석이 함께 돌아가야 뭐가 먹히는지 보입니다.

이 장은 기능을 따로 설명하지 않습니다. 실제 운영에서 어떻게 연결되는지 봅니다.

쉬운 설명

이 장의 핵심은 “붙이기”가 아니라 “이어보기”입니다.

  • Firebase: 사용자 행동과 상태를 저장한다.
  • 광고: 왜 보고, 보고 나서 무엇을 받는지 설계한다.
  • 공유: 눌러보게 만드는 첫 인상을 만든다.
  • 분석: 무엇이 재방문과 수익으로 이어지는지 확인한다.

먼저 지킬 기본

  • Firebase 설정값은 .env.local 로 분리한다.
  • 초기화는 src/firebase/init.ts 한 곳에서만 관리한다.
  • 읽기 실패, 지연, 규칙 오류에 대한 fallback UX 를 준비한다.
  • 저장 성공과 실패를 사용자 메시지와 로그 양쪽에서 확인한다.

놓치기 쉬운 설정

  • Firebase 환경 파일은 Git에 올리지 않고 .gitignore 에서 제외한다.
  • Firestore 와 Storage 는 기본 공개 상태를 믿지 말고 규칙을 직접 잠근다.
  • 허용 도메인은 https://*.apps.tossmini.com, https://*.private-apps.tossmini.com 같은 실제 미니앱 도메인 기준으로 제한한다.
  • 모바일 WebView 환경에서는 서드파티 쿠키 제약이 있을 수 있으므로 파트너사 도메인의 쿠키 로그인에 기대지 말고 토큰 기반 인증을 검토한다.

광고와 OG 기본

  • 광고는 load 와 show 를 나눠 본다.
  • requested 이벤트는 광고 요청이 전달됐다는 뜻으로 이해한다.
  • 보상 지급 시점은 광고 요청이 아니라 실제 완료 조건과 분리해 관리한다.
  • OG 이미지는 1200 × 600 고화질 기준으로 제작한다.
  • OG 에는 과도한 텍스트, 비속어, 정치적 표현, 자극적인 문구를 넣지 않는다.

처음부터 볼 지표

  • 첫 방문 후 첫 카드 열람 전환율
  • 저장 기능 사용률
  • 광고 요청 대비 준비 완료율
  • 광고 준비 완료 대비 시청 완료율
  • 공유 버튼 클릭률과 공유 후 유입률
  • 다음 날 재방문율(D1)

Chapter 07

앱테크는 보상 구조가 전부입니다

앱테크는 사용자가 지금 무엇을 하면 무엇을 얻는지 바로 이해할 때 힘이 생깁니다. 출석, 미션, 광고 보상, 공유 보상은 모두 이 원칙 안에서 움직여야 합니다.

중요한 건 버튼이 아니라 보상 구조입니다. 어디까지는 화면에서 처리하고, 어디부터는 서버로 검증할지 미리 나눠야 합니다. 특히 토스 포인트나 프로모션처럼 실제 보상이 걸린 기능은 서버끼리 안전하게 통신하는 mTLS가 필요합니다.

쉬운 설명

앱테크는 간단한 행동, 바로 느껴지는 보상, 다시 오게 만드는 이유의 조합입니다.

초보자는 광고 버튼부터 떠올리기 쉽습니다. 실제로 더 중요한 건 보상을 어떤 순서로 주는지, 어디서 검증하는지입니다.

가이드 핵심

토스 로그인, 프로모션, 기능성 푸시, 인앱결제처럼 신뢰와 보상이 직접 걸린 기능은 mTLS 기반 서버 간 통신이 필요합니다.
앱인토스 API 사용하기 가이드 핵심 정리

잘 되는 앱테크 루프

  • 첫 10초 안에 보상이 무엇인지 이해된다.
  • 핵심 행동은 한 화면에서 끝난다.
  • 기본 보상과 광고 보너스 보상이 분리돼 있다.
  • 연속 출석, 누적 미션, 시간 제한 보너스로 다음 방문 이유를 만든다.
  • 공유나 친구 초대는 억지가 아니라 보상 구조와 연결된다.
  • 이상 행동 탐지와 지급 로그가 남는다.

화면에서 끝나는 일, 서버가 필요한 일

  • 화면에서 끝낼 수 있는 일: 출석 화면, 미션 진행률, 광고 버튼, 공유 유도, 로컬 보상 시뮬레이션
  • 서버가 필요한 일: 실제 포인트 지급, 추천인 정산, 보상 중복 차단, 운영용 이벤트 집계, 기능성 푸시
  • 꼭 확인할 것: 토스 포인트 프로모션, 로그인, 인앱결제, 서버 연동 기능은 프론트만으로 끝나지 않습니다.

망하기 쉬운 패턴

  • 보상은 큰데 재방문 이유가 약하다.
  • 광고를 먼저 강요해서 첫 경험이 나빠진다.
  • 서버 검증 없이 보상을 지급해 악용이 쉬워진다.
  • 획득 비용과 지급 비용을 계산하지 않는다.
  • 정책과 검수 리스크를 무시한다.

Chapter 08

출시 전에는 검증을 먼저 합니다

화면이 보이는 건 시작입니다. 브라우저, 샌드박스 앱, 실제 토스 앱 WebView에서 다시 봐야 합니다.

배포 전에는 로그와 설정까지 같이 봐야 합니다. 그래야 오류가 나도 빨리 고칠 수 있습니다.

출시 전 꼭 볼 것

  • 브라우저, 샌드박스 앱, 실제 토스 앱 WebView 에서 핵심 흐름을 각각 확인했는가
  • TDS 컴포넌트, 상단·하단 여백, 버튼, 바텀시트가 실제 기기에서 깨지지 않는가
  • Firebase 읽기/쓰기가 규칙과 함께 정상 동작하는가
  • 광고 로드 실패, 공유 실패, 네트워크 오류 시 대체 UX 가 있는가
  • 이벤트 로그가 실제로 남고 있는가

웹 미니앱에서 자주 놓치는 것

  • 브라우저에서 보이는 화면과 샌드박스 앱 WebView 화면을 별도로 확인한다.
  • 샌드박스 앱에서 실행 스킴, 서버 주소, 현재 띄운 개발 서버가 서로 맞는지 본다.
  • 서버에 연결할 수 없으면 granite.config.ts 개발 명령에 --host 옵션이 필요한지 본다.
  • 공유, 광고, 로그인처럼 WebView 문맥이 중요한 기능은 실제 토스 앱 WebView 에서 다시 확인한다.
  • 브라우저에서만 검증하고 끝내지 말고, 샌드박스 앱과 실제 WebView 결과를 최종 기준으로 삼는다.

초보자가 자주 막히는 문제

  • 브라우저에서는 되는데 샌드박스에서 화면이 깨진다.
  • .env.local 키가 누락되어 Firebase 가 초기화되지 않는다.
  • granite.config.ts 의 앱 이름과 실행 스킴이 맞지 않는다.
  • 샌드박스 앱의 서버 주소, 개발 서버 host, 실행 포트를 놓친다.
  • Firestore 규칙 때문에 읽기나 쓰기가 막힌다.
  • 광고 그룹 설정은 했지만 실제 노출 실패 처리를 안 넣었다.
  • OG 이미지는 만들었지만 크기나 카피가 공유 카드에서 약하다.
  • AI가 일반 모바일 웹 코드를 줬는데 그대로 붙여넣었다.
  • 로그를 남기지 않아 어디서 실패했는지 모른다.
  • 배포 후 env 나 서버 함수 설정을 다시 확인하지 않았다.

FAQ와 판매 페이지용 보너스 프롬프트

구매를 고민하는 사람은 기술만 보지 않습니다. 나도 할 수 있는지, 얼마나 걸리는지, 어디서 막히는지가 같이 보여야 합니다.

자주 묻는 질문

Q. 진짜 비개발자도 가능한가?
A. 가능합니다. 다만 복붙하고, 실행하고, 로그를 보는 반복은 필요합니다.

Q. 맥이 없어도 가능한가?
A. 가능합니다. 브라우저, Firebase, 샌드박스 앱, 웹 미니앱 개발·배포 흐름은 Windows 환경에서도 충분히 진행할 수 있습니다.

Q. 어떤 AI 도구를 써야 하나?
A. 도구보다 문서를 어떻게 붙이고, 로그를 어떻게 보여주느냐가 더 중요합니다.

Q. 이 책만 보면 바로 수익이 나나?
A. 아닙니다. 이 책은 수익을 약속하지 않습니다. 대신 수익화 가능성을 검증할 수 있는 구조를 직접 만들게 돕습니다.

Q. 토스 포인트나 프로모션 지급은 화면만으로 가능한가?
A. 아닙니다. 실제 지급이 걸린 기능은 서버와 mTLS 연동이 필요합니다.

Q. WebView 환경에서 쿠키 로그인 방식을 그대로 써도 되나?
A. 조심해야 합니다. 모바일 WebView에서는 쿠키 제약이 있을 수 있어 토큰 기반 방식이 더 안전할 수 있습니다.

Q. 공식 문서만 읽으면 안 되나?
A. 공식 문서는 꼭 필요합니다. 다만 실행 순서와 예외 처리까지 대신 정리해주진 않습니다.

Q. 막히면 어디부터 봐야 하나?
A. 로그, 환경 변수, 앱 설정, 샌드박스 테스트 순서로 보면 대부분 방향이 잡힙니다.
